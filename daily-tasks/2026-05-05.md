## Notes

- **Idempotency key** goes in the request body, not a header — simpler and explicit. Revisit only if an SDK-level convention demands otherwise.
- **Dapper on the hot path** means no `SaveChanges()` on the mission completion write. Explicit SQL, explicit transaction. EF Core stays for profile reads and anything that doesn't need predictable performance.
- **Optimistic concurrency** on the player profile row: version column, check-and-increment in the same transaction. If another request won the race: 409 or retry depending on what the caller should do.
- **Outbox today** can be just the write side — insert the event row in the same transaction as the reward grant. Worker processing can wait for tomorrow if needed.
- Do not spend more than 30 minutes deciding the exact schema. Make a reasonable call, migrate, move on.

---

## Blockers / risks

- **Schema decisions taking too long** — time-box to 30 minutes, then commit.
- **EF Core leaking into Application** — watch for this. If you're tempted to inject `DbContext` into a handler, stop and define an interface instead.
- **Over-engineering the idempotency check** — a simple `EXISTS` query on the idempotency key table is enough. Don't reach for a distributed cache today.
- **Integration test setup friction** — if the test DB setup is slow to configure, stub it out and come back. Don't let test infra block the feature slice.
